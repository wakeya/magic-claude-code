## v0.17.1 更新内容

### Fixed
- **主动清洗保留 tool_reference，修复 kimi deferred 工具调用失效**：`strip_unknown_content_blocks` 开启时，主动清洗原先剥离 `tool_reference`（Claude Code 2.1.x deferred 工具加载标记），导致 kimi-k3 等 deferred MCP 工具调用退化为固定占位 `Bash{command:"true",description:"noop"}`、`thinking` 为空、陷入死循环（模型自述"我一直在发空调用，无法停止"）。2026-07-21 受控实测表明，kimi 三个 Anthropic 兼容端点（moonshot k2.6、coding k2.7、coding k3）现已在 `tool_reference` 引用已定义工具时**接受**该类型；最初（research/2026-05）触发清洗的 `"unsupported content type"` 400 已不复现，残留 400 真因是 `tool_reference` 指向 `tools` 中未定义的工具。`filterContentBlocks` 参数化为 `(preserveToolReference)`：主动清洗保留 `tool_reference`（恢复模型上下文），反应式 `tryRectify` 仍清除它（兜底引用未定义工具的 400）。
- **反应式错误模式匹配覆盖现行 kimi 400 串**：research 时代的 `"unsupported content type"` 已不复现，现 kimi coding 端点报 `"Invalid request Error"`（已被 `hasGenericInvalidRequestPhrase` 覆盖），moonshot 端点报 `"Tool reference 'X' not found in available tools"`。`isUnsupportedContentTypePhrase` 新增 `"tool reference" + "not found"` 收窄匹配（review 反馈：避免误匹配格式/权限/策略类 tool_reference 错误导致反应式清洗掩盖真实错误）。旧 `"unsupported content type"` 用例仍向后兼容。

## v0.17.1 Release Notes

### Fixed
- **Proactive cleanup preserves tool_reference, fixing broken kimi deferred tool calls**: With `strip_unknown_content_blocks` on, proactive cleanup used to strip `tool_reference` (the Claude Code 2.1.x deferred-tool load marker), causing kimi-k3 deferred MCP tool calls to degrade to a fixed `Bash{command:"true",description:"noop"}` placeholder with empty `thinking`, looping forever (the model itself reported "I keep emitting empty calls, can't stop"). A 2026-07-21 controlled live probe shows all three kimi Anthropic-compatible endpoints (moonshot k2.6, coding k2.7, coding k3) now **accept** the type when it references a defined tool; the original `"unsupported content type"` 400 that motivated the cleanup (research/2026-05) no longer reproduces, and remaining 400s are actually caused by a `tool_reference` pointing at a tool absent from `tools`. `filterContentBlocks` is parameterized (`preserveToolReference`): proactive cleanup keeps `tool_reference` (restoring model context), reactive `tryRectify` still strips it (recovers the undefined-tool-reference 400).
- **Reactive error-pattern matching covers current kimi 400 strings**: The research-era `"unsupported content type"` no longer reproduces; the kimi coding endpoint now reports `"Invalid request Error"` (already covered by `hasGenericInvalidRequestPhrase`) and moonshot reports `"Tool reference 'X' not found in available tools"`. `isUnsupportedContentTypePhrase` is extended with `"tool reference" + "not found"` (narrowed per review feedback so format/permission/policy tool_reference errors aren't masked by reactive cleanup). Old `"unsupported content type"` cases stay backward-compatible.

### Docs
- Preserve-tool_reference feature spec / review archive (bilingual): `sdd-docs/features/2026-07-21-preserve-tool-reference/`
- Frontend `strip_unknown_content_blocks` hint and the `StripUnknownContentBlocks` field comment synced: no longer "strips tool_reference / required for strict providers like Kimi"; now "tool_reference is always preserved; enable only if the upstream 400s on other unknown content blocks". Frontend `dist` rebuilt.
- 2026-06-15 proactive cleanup spec / 2026-05-15 research gained a 2026-07-21 correction note recording the kimi upstream behavior change and the original diagnosis's confounding variable (attributing the 400 to the type itself rather than "referencing an undefined tool").
