## v0.16.3 更新内容

### Fixed
- **兼容 Claude Code 2.1.211 新增端点，消除客户端抛异常与遥测噪音**（#17）：CC 2.1.211 引入 `/v1/design/grants`、`/v1/ultrareview/preflight`、`/v1/code/triggers` 等新端点，此前全部命中 fail-closed 拦截返回本地 404——`GET /v1/code/triggers` 失败会让客户端 throw "triggers unavailable" 中断加载流程，`GET /v1/design/grants` 404 会触发 `probe_404_old_server` 遥测噪音。现按端点语义补本地兼容响应：
  - `GET /v1/design/grants` → `200 {"grants":[]}`（空授权列表即禁用 Design 授权）；`POST /v1/design/grants` → `403 write_gate_disabled`（写入门关闭，与 frame deploy 语义一致）
  - `GET /v1/ultrareview/preflight` → `200 {}`（2.1.211 以 preflight 取代 quota，并入同族 `handleEmptyResponse`；`/v1/ultrareview/quota` 保留以兼容旧版本客户端）
  - `GET /v1/code/triggers`（前缀覆盖 `{id}` 子路径）→ `200 {"data":[]}`（空列表避免客户端 throw）；`POST` → `403 write_gate_disabled`；其它方法 405
- 以上端点均不命中模型转发白名单，继续本地处理、不转发上游，无数据泄露面。memory 链路（`/v1/code/local/memory/mounts` 等 5 个端点）经客户端源码验证 404 即其定义的"memory 未启用"正常分支，保持拦截不动，避免过度激活 memory 读写链路。

## v0.16.3 Release Notes

### Fixed
- **Claude Code 2.1.211 endpoint compatibility — no more client throws or telemetry noise** (#17): CC 2.1.211 introduces new endpoints (`/v1/design/grants`, `/v1/ultrareview/preflight`, `/v1/code/triggers`) that previously all hit the fail-closed guard and received local 404s — a failing `GET /v1/code/triggers` made the client throw "triggers unavailable" and interrupt its load flow, and `GET /v1/design/grants` 404s triggered `probe_404_old_server` telemetry noise. Local schema-compatible responses are now served per endpoint:
  - `GET /v1/design/grants` → `200 {"grants":[]}` (an empty grant list disables Design grants); `POST /v1/design/grants` → `403 write_gate_disabled` (write gate closed, same semantics as frame deploy)
  - `GET /v1/ultrareview/preflight` → `200 {}` (2.1.211 replaces quota with preflight; folded into the same `handleEmptyResponse` family; `/v1/ultrareview/quota` is kept for older clients)
  - `GET /v1/code/triggers` (prefix also covers `{id}` subpaths) → `200 {"data":[]}` (empty list avoids the client throw); `POST` → `403 write_gate_disabled`; other methods 405
- None of these endpoints match the model-forwarding whitelist, so they continue to be handled locally and are never forwarded upstream — no data-exposure surface. The memory chain (`/v1/code/local/memory/mounts` and 4 more endpoints) stays blocked at 404: client source review confirmed 404 is its defined "memory not enabled" normal branch, and keeping it blocked avoids over-activating the memory read/write chain.
