# 全站主题系统改造需求文档

**版本：** 0.2
**日期：** 2026-05-21
**状态：** 第二阶段规格审核中

---

## 1. 目标

为管理端前端引入完整的 Light/Dark 主题系统。第一阶段已经通过 `会话记录` 页面验证视觉方向。第二阶段将试点能力提升为全栈主题系统：全局 Header 主题开关、后端持久化管理员偏好、前端本地兜底，以及覆盖整个 Dashboard 的共享主题 token。

参考方向：

1. Light 主题：`https://www.uupm.cc/demo/educational-platform`
2. Dark 主题：`https://www.uupm.cc/demo/developer-tools`

## 2. 成功标准

第一阶段完成的标准：

1. 会话记录页支持显式 Light/Dark 主题开关。
2. 用户选择的主题在刷新后保持。
3. Light 模式具备 educational-platform 参考页的明亮、圆润、平静、亲和的感觉。
4. Dark 模式具备 developer-tools 参考页的现代开发工具质感：深色分层表面、代码编辑器对比度、克制的青色/绿色强调，以及清晰的代码/命令视觉。
5. 会话记录页现有工作流保持正常：项目选择、会话列表滚动、详情展示、用户消息大纲滚动、导出、清理命令提示。
6. 实现中沉淀可复用的主题基础能力，后续推广到全站时不需要再次重写会话记录页。

如果只是替换颜色，但间距、边框、焦点状态、命令/代码块质感仍然割裂，则不算完成。

第二阶段完成的标准：

1. Light/Dark 开关迁移到页面顶部 `AppHeader.vue`，放在右侧操作区，靠近中英文切换和退出登录按钮。
2. Header 中的主题开关成为唯一全局主题入口；移除会话记录页中的临时局部主题开关。
3. 登录后主题偏好由后端持久化，换浏览器或设备时只要使用同一份管理端数据存储，就能保留偏好。
4. `localStorage` 继续作为快速首屏和接口失败兜底，偏好接口返回前可以先应用上次已知主题。
5. 登录页、Header、Tabs、供应商、证书、使用统计、会话记录、弹窗、表单、表格、徽标和代码块都使用同一套语义化主题 token。
6. 切换主题不会重置当前 Tab、已加载数据、打开的弹窗、选中的供应商、选中的会话或可滚动面板状态。
7. 与主题偏好无关的现有 API 行为保持不变。

## 3. 范围

### 3.1 第一阶段：会话记录页试点

主题系统第一阶段只应用到：

1. `SessionBrowser.vue`
2. `SessionDetail.vue`
3. `SessionOutline.vue`
4. 会话清理命令代码块
5. 会话记录页的空状态、加载状态、错误状态
6. 主题切换控件

试点阶段主题开关可以放在会话记录页头部，但设计上必须支持后续迁移到 `AppHeader.vue`，作为全站主题入口。

### 3.2 第二阶段：全栈全站推广

试点确认后，同一套主题系统继续推广到：

1. 登录页
2. 应用头部和导航标签
3. Providers 页面和弹窗
4. 证书页面
5. 使用统计的筛选器、卡片、图表和表格
6. 共享按钮、输入框、选择器、徽标、表格、弹窗和代码块
7. 后端偏好 API 和存储
8. 前端启动时后端偏好与 `localStorage` 的同步

最终主题开关位置是 `AppHeader.vue`，放在右侧操作区，和语言切换、退出登录形成同一组全局操作。

## 4. 主题模型

主题模式：

| 模式 | 含义 |
|------|------|
| `light` | 参考 educational-platform 的明亮 UI |
| `dark` | 参考 developer-tools 的深色开发工具 UI |

第一阶段使用 `localStorage` 保存简单前端偏好。

第二阶段必须采用后端持久化，并保留本地兜底：

1. 管理员主题偏好保存到现有配置存储中。
2. `localStorage` 用于立即首屏渲染，并在偏好 API 失败时作为兜底。
3. 只接受 `light` 和 `dark`。未知、缺失或非法值回退到 `light`。
4. 第二阶段不引入 `system` 模式，管理后台主题保持显式、可预测。

后端偏好 API：

| Method | Path | 请求 | 响应 |
|--------|------|------|------|
| `GET` | `/api/preferences` | 无 | `{ "theme_mode": "light" \| "dark" }` |
| `PUT` | `/api/preferences` | `{ "theme_mode": "light" \| "dark" }` | `{ "success": true, "theme_mode": "light" \| "dark" }` |

两个接口都必须复用现有管理员 session cookie 认证。非法 `theme_mode` 返回 `400`。

后端存储：

1. 在 `config.Config` 中增加 `AdminThemeMode string`。
2. 在现有 SQLite `settings` 表中以 `admin_theme_mode` 保存。
3. 通过 `json:"admin_theme_mode"` 保持旧 JSON 配置兼容。
4. 没有持久化偏好时默认使用 `light`。

主题应通过可复用 CSS 变量或语义化工具类表达，不应在每个组件里散落大量一次性硬编码颜色。第二阶段应在 document root 或顶层 app root 上使用稳定属性（例如 `data-theme="light|dark"`）全局应用当前主题。

需要覆盖的语义 token 组：

| Token 组 | 用途 |
|----------|------|
| 应用背景 | 页面级背景 |
| 表面 | 主卡片、面板、弹窗 |
| 抬升表面 | 选中卡片、弹出层、焦点区域 |
| 边框 | 普通边框和强调边框 |
| 文本 | 主文本、次级文本、弱化文本 |
| 强调色 | 主要操作和选中状态 |
| 成功/用户消息 | 用户消息块 |
| 警告 | 清理提示和风险提示 |
| 代码 | 命令/代码编辑器表面和 token 颜色 |

## 5. 视觉方向

### 5.1 Light 主题

Light 模式应满足：

1. 使用明亮的偏白/偏蓝背景，而不是单调灰色。
2. 使用白色半透明或柔和面板，搭配轻微蓝色边框。
3. 圆角可以更友好，但必须保持专业和克制，不能变成过大的营销卡片。
4. 蓝色用于主要操作和选中状态，绿色用于用户消息高亮。
5. 保持管理后台适合反复使用的信息密度。
6. 保留当前项目使用的 Outfit 字体和可读性。

### 5.2 Dark 主题

Dark 模式应满足：

1. 使用深 navy/slate 背景，不使用纯黑。
2. 使用层次清晰的深色表面和克制边框。
3. 青色/天蓝色用于选中状态和主要操作。
4. 用户消息绿色块在暗色模式下应清晰可读，但不能发光刺眼。
5. 清理命令以深色代码编辑器风格展示，token 高亮清晰。
6. 文本、焦点状态和控件都要满足可访问的对比度。

## 6. 交互要求

1. 第一阶段主题开关在会话记录页上显式可见。
2. 第二阶段主题开关在顶部 Header 中显式可见，位置靠近语言切换和退出登录。
3. 切换主题不能重置当前项目、当前会话、滚动位置、已加载详情状态、当前 Tab、打开的弹窗或供应商表单状态。
4. 主题偏好刷新后保持，并在已认证会话中由后端持久化。
5. Light/Dark 下键盘焦点状态都必须清晰可见。
6. 复制、导出、清理命令提示等现有交互保持不变。
7. 如果后端偏好更新失败，UI 保持用户刚选择的本地主题，写入 `localStorage`，并在下次应用加载时重试或重新同步。

## 7. 非目标

1. 第一阶段不改造后端 API。
2. 第一阶段不重设计所有 Dashboard 页面。
3. 不为了主题偏好引入多用户角色或独立用户账户体系。
4. 不引入重型组件库。
5. 不在管理后台增加动画很重或营销风格的 hero 页面。
6. 不使用参考站点的外部运行时资源。
7. 第二阶段不增加 `system` 主题模式。

## 8. 兼容性

1. 试点必须保持现有 Vue 3 + Tailwind CSS v4 技术栈。
2. 主题样式必须兼容当前嵌入式前端构建方式。
3. 会话记录页在桌面和较小屏幕上都必须可用。
4. Dark 模式不应破坏导出 HTML；导出样式仍由后端导出模板控制，除非后续阶段另行明确。
5. 偏好 API 必须使用和现有 Dashboard API 相同的管理员认证中间件。
