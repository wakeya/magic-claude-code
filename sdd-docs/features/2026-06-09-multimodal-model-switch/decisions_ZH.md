# 多模态模型切换 — 决策记录

## D-001：使用单一多模态模型 ID

**日期：** 2026-06-09
**状态：** accepted

**背景：** 多模态切换只用于“请求含非文本内容时文本模型不兼容”的供应商场景。按原始模型分别配置多模态映射会增加 UI 和配置复杂度。

**决策：** Provider 只配置一个 `multimodal_model`。当请求中包含图片、PDF、音频或视频等非文本内容时，统一切换到该模型。

**影响：** 配置简单，排查清晰。缺点是同一 Provider 下无法针对不同原始模型选择不同多模态模型。

**重新评估条件：** 同一供应商需要同时维护多个多模态能力层级，例如 Opus 请求切到高阶视觉模型、Sonnet 请求切到低成本视觉模型。

## D-002：开关命名为“多模态切换”

**日期：** 2026-06-09
**状态：** accepted

**背景：** “遇到非文本内容时切换模型”语义准确但过长；“支持多模态”又容易误导用户，以为是在声明供应商能力。

**决策：** 前端开关命名为 `多模态切换`，字段命名为 `多模态模型 ID`。

**影响：** UI 更简洁，同时保留“代理切模型”而非“供应商支持能力”的语义。

**重新评估条件：** 后续 Provider 配置页增加更复杂的能力矩阵，需要更明确区分“供应商能力”和“代理策略”。

## D-003：默认关闭，兼容供应商不受影响

**日期：** 2026-06-09
**状态：** accepted

**背景：** GLM-5.1 等供应商可能已兼容含图片的请求，即使模型名看起来是文本模型。全局自动切换会破坏这类供应商的正常行为。

**决策：** 多模态切换默认关闭。只有用户显式开启且填写模型 ID 时才可能覆盖模型映射。

**影响：** 保持现有 Provider 行为稳定。Mimo 这类不支持图片输入的供应商需要用户主动配置。

**重新评估条件：** 后续能从供应商能力接口可靠获取模型多模态支持情况。

## D-004：递归检测请求体中的非文本内容

**日期：** 2026-06-09
**状态：** accepted

**背景：** 实际故障中的图片不是用户顶层消息，而是截图工具返回的 `tool_result.content` 内嵌图片。只检查顶层消息会漏检。

**决策：** 在 `messages` 和 `system` 下递归扫描数组和对象，匹配明确的内容块类型或 `source.media_type`。

**影响：** 能覆盖截图、PDF 和其他工具返回的非文本内容。扫描成本随请求体大小线性增长，但复用现有 JSON 解析，常规请求影响可忽略。

**重新评估条件：** 发现新的非文本内容格式不在当前匹配规则内。

## D-005：不修改协议头

**日期：** 2026-06-09
**状态：** accepted

**背景：** 本次故障由上游模型不支持 image input 导致，而非 Anthropic Beta Header 不兼容。过早修改协议头可能破坏其他供应商。

**决策：** 第一版只覆盖 `model` 字段，不修改 `Anthropic-Beta`、`Anthropic-Version` 或其他请求头。

**影响：** 变更范围小，风险低。若某多模态模型仍因 Header 报错，需要另行设计 Provider 级协议兼容策略。

**重新评估条件：** 配置多模态模型后，上游继续因 Beta Header 或协议字段返回兼容性错误。
