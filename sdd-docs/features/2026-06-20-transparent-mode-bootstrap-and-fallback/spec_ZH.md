## 整体分析（源站分析）

### 当前项目状态

项目已经在启动时生成本地 CA 和服务端证书，将代理绑定到 `443`，并打印手动配置指引。当前流程本身是“透明模式优先”：

- 预期 `api.anthropic.com` 解析到 `127.0.0.1`
- 本地 CA 写入到 `data/ca.crt`
- 用户需要手动安装 CA，并手动导出 `NODE_EXTRA_CA_CERTS`

这套流程能工作，但对于用户要的部署体验来说，手工步骤太多。当前启动路径不会自动修改 hosts，也不会自动处理信任根；当系统权限或平台策略阻止这些操作时，也没有结构化的降级出口。

### 这个功能要解决什么

用户希望运行时在可能的情况下自动完成原本需要手工做的配置：

1. 添加 `127.0.0.1 api.anthropic.com`
2. 信任生成的本地 CA
3. 立即启动代理
4. 如果主机无法修改，则切换到更低权限的连接模式，而不是把用户卡死在失败状态

同时，自动化失败时必须打印可读、且符合当前系统语言的指引：

- 中文系统默认中文输出
- 其他语言默认英文输出

### 必须遵守的模式优先级

本功能引入明确的入口优先级：

| 优先级 | 模式 | 入口 | 能否拦截硬编码 `api.anthropic.com` 流量 | 权限要求 | 作用 |
| --- | --- | --- | --- | --- | --- |
| 1 | 透明模式 | `hosts` + `443` TLS | 可以 | 需要主机修改权限和本地 CA 信任 | 兼容性最高，当前默认能力 |
| 2 | 隧道模式 | `HTTPS_PROXY` + CONNECT MITM | 大体可以 | 不修改 hosts；运行时必须信任本地 CA | hosts/证书安装被阻止时的最佳降级 |
| 3 | 网关模式 | `ANTHROPIC_BASE_URL=http://127.0.0.1:17487` | 不可以 | 不修改 hosts；不依赖系统 CA；不需要 `443` | 最低权限的后备入口 |

顺序非常重要：

- 透明模式必须先尝试，因为它兼容性最好，也最能覆盖硬编码的官方端点。
- 隧道模式必须排在第二，因为它在不改 hosts 的情况下仍有机会接住硬编码请求。
- 网关模式必须最后，因为它无法拦截硬编码端点，只能覆盖尊重 `ANTHROPIC_BASE_URL` 的客户端。

### 参考来源：现有低权限入口设计

临时的 `claude-code-proxy` 规格已经定义了一个兼容的低权限拆分：

- `ANTHROPIC_BASE_URL` 模式，覆盖模型路径
- `HTTPS_PROXY + CONNECT MITM` 模式，覆盖范围更广

本功能应该复用这个思路，而不是另起一套互相打架的降级方案。不同之处在于，这里关注的是 **MCC 启动时的自动引导和自动降级选择**，不是替换透明模式。

### 日志与降级要求

自动引导失败时，程序必须同时做到以下几件事：

1. 记录明确的失败原因
2. 说明当前缺少的能力，例如：
   - 不能修改 hosts
   - 不能安装或信任 CA
   - 不能持久化环境变量
3. 打印下一推荐模式及其启动方式
4. 动态打印当前 `ca.crt` 路径
5. 如果失败状态没有变化，不要每次启动都重复刷一大段相同指引

### 首次运行环境引导

本地二进制安装在第一次成功运行时，必须尝试把二进制所在目录变成未来任意工作目录都能自动发现的 MCC 根目录。

必须满足的行为：

- 解析真实可执行文件位置，而不是当前 shell 工作目录
- 把可执行文件目录视为这台机器上的 MCC 安装根目录
- 持久化一个适合当前用户的环境项或启动配置项，使后续运行能自动定位 MCC 根目录
- 使用这个根目录定位捆绑的 CA 证书，不要求用户手动导出或复制证书路径
- 如果运行时无法持久化环境变更，要记录本地化失败信息，并退回手动说明或低权限模式

环境引导只是便利层，不应被视为查找 CA 的唯一方式；如果机器策略阻止持久化，也不能因此阻塞启动。

### Docker 边界

Docker 是特殊情况：

- 容器内可以生成证书并启动服务
- 容器不能可靠地修改宿主机的 hosts 文件或系统 CA 信任库
- 如果存在宿主机侧 helper，可以调用它
- 如果没有 helper，运行时不能假装宿主机已经被修改，必须记录失败并打印后备指引

这个边界必须在设计中保持清楚。Docker 不能被包装成“容器自己就能改宿主机”的假象。

## 需求

### 交付物

1. 启动时，服务必须在代理启动前自动尝试透明模式配置：
   - 确保本地 CA 存在
   - 尝试让 `api.anthropic.com` 解析到 `127.0.0.1`
   - 尝试用平台合适的方式安装或信任生成的 CA
   - 即便其中某一步失败，也不能因此阻止代理启动
2. 引导和降级日志必须遵循当前系统语言：
   - 中文 locale 输出中文日志和指引
   - 其他语言默认输出英文日志和指引
3. 如果透明模式因为权限不足或系统策略失败，运行时必须按优先级选择下一种可用模式：
   - 先隧道模式
   - 再网关模式
4. 如果所有自动尝试都失败，运行时必须打印本地化的手工兜底指南，至少包含：
   - 失败原因
   - 缺失能力
   - 当前 `ca.crt` 路径
   - 下一推荐模式和对应命令
5. 运行时不能因为引导失败就阻塞代理启动。
6. Docker 运行不能在没有 helper 成功的情况下声称宿主机已被修改。
7. 后备指引要足够稳定和有状态，避免在同一个失败状态下每次启动都重复刷同一大段文本。
8. 顶部 header 必须提供明显的模式入口区域，让用户第一眼就能看到并检查透明模式、隧道模式和网关模式。
9. 本地二进制第一次成功运行时，必须尝试持久化一个可发现的 MCC 根目录，这样之后从任意工作目录启动都能自动找到证书路径。
10. 设计必须复用当前证书生成逻辑和现有 locale 选择方式，不要再引入一套独立的本地化系统。

### 模式语义

#### 透明模式

透明模式是当前的完整兼容路径，必须保持第一尝试，因为它是唯一能够完整保留 `api.anthropic.com` 伪装效果的模式。

预期自动化目标：

- hosts 文件更新
- CA 信任安装或信任提升
- `443` 绑定准备

如果任何一步不能自动完成，运行时应该记录失败并进入降级。

#### 隧道模式

隧道模式是当 hosts 层无法修改时的兼容后备方案。它应依赖：

- 进程级代理环境变量
- CONNECT MITM 支持
- 运行时对本地 CA 的信任

隧道模式仍然比网关模式更强，因为它仍有机会拦住硬编码官方端点。在环境支持代理变量和运行时 CA 信任时，它应该优先于网关模式被推荐。

#### 网关模式

网关模式是最低权限后备入口。只有在以下情况才应推荐它：

- 透明模式无法完成
- 隧道模式无法使用，或者在当前运行时场景下没有意义

网关模式只覆盖尊重 `ANTHROPIC_BASE_URL` 的客户端。它必须被呈现为有限后备，而不是透明模式或隧道模式的替代品。

### 顶部 Header 模式入口

顶部 header 必须包含一个明显的模式入口区域，作为三种模式的主要发现入口。

必须满足的行为：

- 在 header 内直接显示当前推荐优先级：`透明模式 > 隧道模式 > 网关模式`
- 在 header 内展示三个可区分的模式按钮或胶囊标签
- 点击 header 入口后打开详细模式说明弹窗
- header 入口只负责展示和发现，不负责立即重写运行配置
- 该区域必须足够醒目，让用户无需滚动就能看到

header 不应放入长篇启动命令块。那部分内容必须单独放置，避免把顶部区域做得过高、过杂。

### 手工指引规则

失败时打印的手工指引必须基于真实运行时状态动态拼接：

- `ca.crt` 路径必须来自当前数据目录
- hosts 映射必须指向 `api.anthropic.com`
- 环境变量命令必须按平台区分
- 指引内容必须和当前选中的后备模式一致，不能夹带无关步骤

### 可执行文件目录证书解析

运行时必须优先把 `mcc` 可执行文件所在目录作为本地二进制安装的证书发现来源。

含义如下：

- 捆绑的 CA 证书通过可执行文件所在目录来发现，而不是通过调用者当前工作目录来发现
- 只要首次运行的环境引导成功，从其他工作目录启动 `mcc` 也应该能解析出正确的 CA 路径
- README 必须告诉用户第一次运行时使用管理员权限，以提高环境持久化成功率
- 如果没有管理员权限，运行时必须退回到手动导入说明或隧道/网关模式，而不能阻塞启动

### 非目标

1. 不把透明模式替换成网关模式。
2. 不引入内核级劫持、WFP、TUN/TAP、WinDivert、Npcap 或 DLL 注入。
3. 不在没有 helper 或等效特权路径的情况下，从 Docker 悄悄修改宿主机。
4. 本期不增加新的图形向导。
5. 不声称网关模式可以拦截硬编码的 `api.anthropic.com` 流量。
6. 不因为引导无法完成就让启动失败。
7. 不要求用户在首次运行环境引导成功后，仍然手动复制证书路径。

### 任务 2：透明模式自动引导执行器

#### 需求

**Objective（目标）** — 在启动时自动尝试当前手工的透明模式配置步骤。

**Outcomes（成果）** — 运行时能创建或加载 CA，尝试写 hosts，尝试安装/信任 CA；其中某一步失败时，仍继续启动代理。

**Evidence（证据）** — 使用 mock OS adapter 的测试能够覆盖成功路径和权限拒绝路径。

**Constraints（约束）** — 执行器必须明确记录它做到了什么、没做到什么，不能伪造成功。

**Edge Cases（边界）** — hosts 文件只读；CA store 被策略锁定；某一步成功但后面仍需要降级；Docker 下根本不能改宿主机。

**Verification（验证）** — 成功、部分失败和完全失败的单元测试。

#### 计划

1. 新增 `internal/bootstrap/` 包，定义结果模型，分别记录 hosts、CA 信任和环境持久化结果。
2. 把该包拆成更小的单元，例如 `executor`、`capabilities`、`instructions`、`state`，让启动、报告和格式化彼此隔离。
3. 在 `cmd/server/main.go` 中，把 bootstrap 放在 `certManager.EnsureCA()` 和 `EnsureServerCert()` 之后、代理和管理服务 goroutine 启动之前。
4. 为 hosts 修改、CA 信任安装、环境持久化分别增加 OS 适配器。
5. 让每个适配器返回带类型的成功/失败信息，而不是只返回一条总错误。
6. 将 bootstrap 状态持久化到 data 目录，避免同一个失败在每次启动时都打印完整长指引。
7. 保持代理和管理服务启动不变，bootstrap 失败不能中止进程。

- [x] 成功引导会报告成功。
- [x] 权限失败会被捕获并报告。
- [x] 引导失败后代理仍然能启动。
- [x] bootstrap 状态能被复用，避免重复长日志。

### 任务 4：Docker 与 helper 边界处理

#### 需求

**Objective（目标）** — 让 Docker 行为诚实且明确，同时在存在宿主机 helper 时仍允许自动引导。

**Outcomes（成果）** — Docker 启动时可以生成 CA 并启动服务，但只有在有特权 helper 真正成功时，才会把宿主机修改标记为成功。

**Evidence（证据）** — 没有 helper 的 Docker 运行会打印本地化后备指引，而不会暗示宿主机已被修改。

**Constraints（约束）** — 容器本地动作允许；宿主机修改不能伪造。

**Edge Cases（边界）** — helper 缺失；helper 无法获得权限；容器可以写 CA，但宿主机没有变化；helper 只成功写了 hosts，没成功写 CA 信任。

**Verification（验证）** — 有/无 helper 的 Docker 手工运行验证。

#### 计划

1. 在 `cmd/server/main.go` 中识别 Docker 启动场景，并区分容器内引导和宿主机引导。
2. 只允许 host-side helper 承担宿主机修改和信任库操作。
3. 如果没有 helper，则记录明确的 Docker 限制失败并继续启动。
