## D-005：反应式恢复替代主动预处理

**日期：** 2026-05-18
**状态：** 已采纳

**背景：** 通过研究 cc-switch（Rust 代理）证实，Anthropic 兼容供应商（GLM、MiniMax、Kimi、DeepSeek、Qwen）在正常情况下可以直接透传请求，无需预处理。错误仅在特定条件下发生（tool 校验、thinking 签名、长对话块类型）。为特定供应商做主动预处理增加复杂度，且有破坏正常供应商的风险。

**决策：** 改为反应式错误恢复。所有请求直接透传。当上游返回 400 且匹配已知错误模式时，执行针对性清理并重试一次。

**影响：**
- 正常供应商（GLM、Anthropic 官方）收到零修改的请求。
- 清理仅在确认出错后才执行。
- 错误模式跨供应商共享 — MiniMax 和 Kimi Code 受益于同一机制。
- 机制可扩展：新增错误模式无需供应商配置。

**重新评估条件：** 某供应商持续在首次请求失败，重试增加不可接受的延迟。此时考虑添加可选的主动清理开关。

## D-006：单次重试，不多次尝试

**日期：** 2026-05-18
**状态：** 已采纳

**背景：** 400 错误的多次重试会显著增加延迟，并浪费供应商配额。清理策略是确定性的 — 如果第一次重试失败，同样的清理第二次重试也会失败。

**决策：** 每个客户端请求最多重试一次。

**影响：** 实现简单。若清理策略无法修复问题，用户看到的错误与之前相同。

**重新评估条件：** 存在多个独立的清理策略可以顺序尝试。

## D-007：仅在 /v1/messages 端点重试

**日期：** 2026-05-18
**状态：** 已采纳

**背景：** 代理转发多种 HTTP 路径。错误恢复仅对 Claude Code 的消息端点有意义。

**决策：** 仅在 `/v1/messages` 和 `/anthropic/v1/messages` 激活重试。

**影响：** 其他端点（健康检查、模型列表）不受影响。

**重新评估条件：** 其他端点出现可恢复的 400 错误。

## D-008：Tool 清理仅移除非必要元数据

**日期：** 2026-05-18
**状态：** 已采纳

**背景：** Tool 定义影响模型行为。移除核心字段（name、description、properties）会降低 tool 调用准确度。

**决策：** Tool 清理仅移除：
1. `cache_control`（Anthropic 缓存提示）。
2. JSON Schema 元数据（`$schema`、`$id`、`$comment`）。
3. 空/缺失的 `input_schema` 填充为 `{"type":"object","properties":{}}`。
4. `additionalProperties`（部分供应商拒绝此字段）。

核心字段（name、description、type、properties、required）永远不被修改。

**影响：** 对 tool 调用准确度的风险极低。清理仅移除模型不用于 tool 选择/调用的字段。

**重新评估条件：** 某供应商拒绝当前保留的字段（如 schema 中的 `oneOf`、`anyOf`）。
