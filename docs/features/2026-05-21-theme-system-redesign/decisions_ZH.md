# 全站主题系统改造决策记录

## D-001：先用会话记录页验证主题系统

**日期：** 2026-05-21
**状态：** accepted

**背景：** 最终目标是完整的全站 Light/Dark 主题系统。如果一次性重做所有 Dashboard 页面，会触及大量组件，视觉审核、回归测试和回滚成本都会变高。

**决策：** 第一阶段先以会话记录页作为试点。会话记录页包含导航、卡片、详情面板、代码块、弹窗和消息内容，视觉密度较高，适合作为主题系统的验证场。

**影响：** 第一阶段可以验证视觉方向，同时避免全站重写。主题基础能力仍必须按后续复用来设计。

**重新审视：** 当会话记录页试点被确认后，再把同一套系统迁移到 `AppHeader.vue` 和共享组件。

## D-002：使用显式主题开关

**日期：** 2026-05-21
**状态：** accepted

**背景：** 只跟随系统主题可能让用户感到不可控。用户已明确要求提供显式开关。

**决策：** 会话记录页试点阶段提供可见的 Light/Dark 主题开关。第一阶段使用本地持久化。控件设计上要支持后续迁移到应用头部，成为全站主题开关。

**影响：** 主题切换是明确、可测试的。第一阶段不需要后端持久化。

**重新审视：** 当多管理员或共享偏好需要服务端保存时，再引入后端持久化。

## D-003：参考站点只作为风格方向，不作为资源来源

**日期：** 2026-05-21
**状态：** accepted

**背景：** 参考页面提供了有效的视觉方向，但管理后台的信息密度、工作流和品牌需求不同。

**决策：** 只借鉴参考页面的风格方向，不复制外部运行时资源、不照搬页面结构、不引入营销页式构图。视觉语言需要适配现有管理后台工作流。

**影响：** 最终效果应有参考风格的气质，但仍然是实用的管理工具界面。

## D-004：最终主题开关迁移到应用 Header

**日期：** 2026-05-21
**状态：** accepted

**背景：** 会话记录页试点把开关放在会话页面内，是为了快速验证视觉方向。最终产品中，主题选择应该是全局应用偏好，而不是某个页面自己的局部控件。

**决策：** 全站推广阶段将 Light/Dark 开关迁移到 `AppHeader.vue`，放在右侧操作区，靠近语言切换和退出登录按钮。当 Header 开关控制全站后，移除会话记录页中的临时局部开关。

**影响：** 用户可以在任意 Dashboard Tab 中切换主题。主题控件成为全局应用外壳的一部分，会话记录页样式需要继承全局主题，而不是维护独立的局部主题来源。

## D-005：主题偏好后端持久化，localStorage 兜底

**日期：** 2026-05-21
**状态：** accepted

**背景：** `localStorage` 足够支撑试点，但目标全栈系统应当在使用同一份管理端数据存储时，跨浏览器和跨设备保留偏好。

**决策：** 管理员主题偏好保存到后端配置存储中，并通过受认证保护的偏好接口读写。`localStorage` 继续作为快速首屏和失败兜底。

**影响：** UI 可以在加载初期立即应用本地主题，然后再和后端偏好对齐。登录后以后端为事实来源；当偏好 API 临时不可用时，本地持久化保证界面仍然可用。

## D-006：只支持显式 `light` 和 `dark`

**日期：** 2026-05-21
**状态：** accepted

**背景：** 跟随系统的 `system` 模式会增加启动、同步和测试复杂度。用户已经要求显式开关。

**决策：** 第二阶段只支持 `light` 和 `dark`。非法值或缺失值回退到 `light`。

**影响：** 主题状态简单、明确、可预测。未来只有在全站推广后用户明确需要时，才重新考虑 `system` 模式。
